# Ideas for Better Treesearch Training Data

## Problem

When evaluating DPVT models on tree search intermediates, performance (AUROC)
degrades as the tree search progresses toward optimal trees. Specifically, once
the parsimony score improves dramatically, the model's ability to classify MP vs
non-MP edges gets worse.

### Root cause: distribution mismatch

The training data and treesearch test data have fundamentally different
distributions of non-MP edges:

- **Training data** is generated by taking MP trees from larch and perturbing
  them to a **fixed target non-MP fraction** (~10% by default via
  `spr_target_non_mp_proportion = 0.1` in
  `dpvtex/larch/scripts/generate_configs.py`). Every training tree has roughly
  the same proportion of non-MP edges.

- **Treesearch test data** consists of intermediate trees from an actual MP tree
  search. The non-MP fraction varies continuously: starting trees (random or NJ)
  have a **high** fraction of non-MP edges, and trees near the end of the search
  have a **very low** fraction (approaching zero as the search converges to an
  MP tree).

The model has never seen trees with very few non-MP edges during training, so it
performs poorly in exactly the regime that matters most: trees that are almost
optimal but have a handful of remaining non-MP edges.

### Existing approach: `treesearch_mimic`

The existing `treesearch_mimic` edge distribution
(`dpvtex/larch/scripts/extract_data_from_hdag.py`, lines 304-352) was designed
to address this by creating a mix of trees at different non-MP fractions:

- 1/2 random trees (most edges non-MP) - simulates early search (the tree search
  starts from a random tree, so this is a reasonable component)
- 1/4 trees perturbed to full target proportion - simulates mid search
- 1/4 trees with target drawn uniformly from [0, target] - provides variety

This has not been tested on treesearch evaluation yet. One potential concern is
that the **low non-MP regime is underrepresented**: only a small slice of the
uniform [0, target] quarter will produce trees with near-zero non-MP fractions,
which is exactly the regime where we observe the performance drop.

### Labeling concern: phangorn can outperform larch

In some cases, the best tree found by phangorn has a better parsimony score than
any tree in the larch DAG. Since edge labels are assigned by checking against
the larch DAG (via `assign_edge_labels()` in
`dpvtex/larch/scripts/extract_data_from_hdag.py`), edges in a better-than-larch
tree may be labeled as non-MP even though the tree is objectively superior.

This has two implications:

1. **Evaluation artifact**: Part of the AUROC degradation near the end of the
   search may not reflect actual model failure. The model might be predicting
   edges correctly, but the ground truth labels are wrong for trees that surpass
   larch's optimum.
2. **Training data quality**: Any approach that uses treesearch intermediates as
   training data (Idea B) would include these mislabeled trees, potentially
   teaching the model incorrect patterns.

Possible mitigations:

- Start larch from the best phangorn tree (instead of building the DAG
  independently). This ensures the larch DAG contains trees at least as good as
  the best search intermediate, making all labels correct.
- Run larch for more iterations to find a better optimum before labeling
- Filter out trees with better parsimony score than the larch DAG before
  labeling
- Investigate how often this happens and how many edges are affected

Note: starting larch from phangorn's tree ties the MP definition to phangorn's
search neighborhood, but this is inherent to evaluating phangorn tree searches
and not a new bias introduced by this choice.

## Ideas

### A. Target the hard regime: near-MP trees (try first)

**Description**: Generate a training set specifically focused on trees with very
low non-MP fractions (1-5%). This is the cheapest experiment to check whether
the model can learn to classify near-MP trees at all. If it can't, the more
involved approaches (B-D) won't help either.

**Expected benefit**: Answers the key question: is the near-MP regime learnable?
Also directly addresses the failure mode.

**Implementation**:

- Use `perturb_tree_with_spr_target()` from
  `dpvtex/larch/scripts/tree_perturbation.py` with low
  `spr_target_non_mp_proportion` (0.01-0.05)
- May need `spr_radius=1` to ensure individual SPR moves are small enough to
  achieve very low non-MP fractions
- Generate a dedicated training set and evaluate on treesearch data

**Considerations**:

- With very low targets, many trees may end up with 0% non-MP edges if the
  perturbation can't find valid SPR moves that introduce exactly the right
  number of non-MP edges
- Small perturbations may not create diverse enough training signal
- If this works, combine with broader approaches (C or D) for full coverage

### B. Seed larch with phangorn's best tree

**Description**: Start the larch run from the best tree found by phangorn
(rather than building the DAG independently). The resulting DAG contains MP
trees from the same region of tree space that phangorn converges to, so
perturbations of those trees better represent what the model encounters during
treesearch evaluation. This also resolves the labeling concern above.

**Expected benefit**: Medium-high. Improves both label correctness and the
relevance of synthetically perturbed training data. Complements ideas C-D and
resolves the labeling concern for idea D.

**Implementation**:

- Modify the `log_tree_searches` pipeline to pass phangorn's best tree as input
  to larch (currently larch builds its DAG independently from the FASTA)
- The `run_larch_usher` rule in `dpvtex/log_tree_searches/Snakefile` would need
  to accept the phangorn output tree as a starting point

**Considerations**:

- Ties the MP definition to phangorn's search neighborhood (see note above)
- Requires a pipeline change to connect phangorn output to larch input

### C. Generate training data with diverse non-MP fractions

**Description**: Instead of generating all training data at a single target
proportion (10%), cover the full range of non-MP fractions seen during tree
search. The low non-MP regime is particularly important since that's where the
model currently fails.

**Expected benefit**: Medium-high. Covers the full range of non-MP fractions,
including the critical low-fraction regime.

**Implementation options**:

- **Multiple pipeline runs**: Use the existing `-p` flag in `generate_configs.py`
  to generate training data at several targets (e.g., 1%, 5%, 10%, 20%, 50%)
  and combine the resulting pickle files
- **Tune `treesearch_mimic`**: First test the current `treesearch_mimic` as-is.
  If the low non-MP regime is still underrepresented, adjust the split in
  `generate_perturbed_trees_with_labels()` in
  `dpvtex/larch/scripts/extract_data_from_hdag.py` (lines 304-352), e.g. by
  adding a dedicated near-MP quarter or sampling targets from a distribution
  skewed toward zero
- Could analyze existing treesearch test data to empirically inform the target
  distribution

**Considerations**:

- Very low targets (1-2%) may be hard to achieve with SPR moves, depending on
  `spr_radius` - a single SPR on a small tree might already exceed 1%
- Wait for `treesearch_mimic` test results before tuning it
- Synthetic perturbations may not capture all structural properties of real
  search intermediates

### D. Use real treesearch intermediates as training data

**Description**: Run the `log_tree_searches` pipeline on training alignments
(not just test alignments) and use the resulting labeled intermediate trees
directly as training data. This gives the model exposure to the exact
distribution of non-MP edges it will encounter during evaluation.

**Expected benefit**: High. The model would train on trees that look exactly
like what it's tested on, including the critical low non-MP regime near the end
of searches.

**Implementation**:

- The `dpvtex/log_tree_searches/Snakefile` pipeline already handles the full
  workflow: run phangorn tree search, run larch for the MP DAG, label edges via
  `create_testing_data.py`
- Would need to run this on training alignments (currently only run on test
  alignments)
- The `add_to_dataset_nicknames.py` script handles registering the output
  datasets
- Main work: running the pipeline on training data and configuring the training
  workflow to use the output

**Considerations**:

- Tree searches are expensive (R phangorn + larch per alignment)
- Training and test data should use different alignments to avoid data leakage
- The number of trees per search varies; may need to balance across alignments
- Trees early in the search may dominate the dataset (searches log many early
  trees before converging)

---

## Suggested issue structure and order

These should be split into three separate issues:

1. **Idea A** (feasibility experiment) - cheap and answers whether the model can
   learn the near-MP regime at all. If it can't, the other ideas won't help.
2. **Idea B** (seed larch with phangorn's tree) - can start in parallel with A.
   Improves label correctness and data relevance for everything downstream.
3. **Ideas C or D** (broader coverage) - pick based on results from A and B.
